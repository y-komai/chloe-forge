Appendix A ハーネスエンジニアリング実例

自分の構成（CLAUDE.md ベース）
使えるツール
GitHub / Discord / Web検索 / ローカルファイル
止まる条件
外部への送信・投稿は確認を取る
記憶の範囲
memory/ ファイル群（長期・セッション別）
禁止操作
破壊的変更（rm -rf 等）は承認要
世間の事例——みんな同じことをやってる
Cursor Rules
→ コード生成のスタイル・禁止パターンを制約
GitHub Copilot Instructions
→ リポジトリの文脈・命名規則・禁止ライブラリを制約
Devin 組織設定
→ アクセス可能なリソース・外部API呼び出し・承認フローを制約
Claude Code（CLAUDE.md）
→ ツール権限・行動指針・記憶ポリシーを制約

ツールは違う。でも「何をどこまでやらせるか」を設計している点は同じ

Appendix C OpenClaw 構成の詳細

本編スライド3：外から見た構成
  • 💬 Discord入口。24時間常駐、話しかけると反応
  • 🧠 記憶会話履歴を保持、日報を自動まとめ
  • 🛠 ツール実行GitHub 操作、ファイル読み書き
  • 🔗 AI 連携他の AI に指示、A2A オーケストレーション
実際の内部構成
  • GatewayDiscord/Slack からのメッセージ受信・送信
  • OrchestratorClaude が「次に何をするか」を判断。ループの核
  • Memory短期（会話文脈）＋長期（ファイルベース）の2層
  • Heartbeat定期実行。日報生成・ログ保存などを cron 的に動かす
  • Tool ExecutorGitHub / Web / ファイル操作を安全に実行
  • Sub-agent BridgeDevin 等の外部 AI エージェントへ指示を橋渡し
