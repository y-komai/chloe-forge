WebサービスエンジニアのAIエージェント運用自動化

あなたの午前3時はどんな感じですか？

PagerDutyが鳴る。目が覚める。スマホでSlack確認。ラップトップ開く。
ログ確認して、メトリクス確認して、関連サービス確認して——
30分後、「やっぱり再起動するか」とコマンド打つ。

この作業——手順書に書けますか？

書けるなら、エージェントが代わりにできるかもしれない。

初動対応の正体

  • 1
    アラートの確認
    どのサービス？どのメトリクス？閾値はいくつ？
  • 2
    ログの調査
    直前のエラーログ・スタックトレース・関連サービスの状態
  • 3
    影響範囲の特定
    どのユーザー？どのエンドポイント？どれくらい？
  • 4
    判断と対応
    再起動で直るか？ロールバックが必要か？エスカレーションか？

1〜3はルーティン。4だけが判断。

PagerDutyが消えるのではなく、人間が呼ばれる頻度が下がる

4つのエージェントの役割分担

Agent 1
アラート受信
PagerDuty/Webhookを受け取り、アラートの内容を構造化して次エージェントへ渡す
→ Webhook受信 / PagerDuty API
Agent 2
ログ・メトリクス調査
エラーログ・スタックトレース・関連サービスの状態を収集してレポートを作成
→ Datadog / CloudWatch MCP連携
Agent 3
トリアージ・修復試行
調査結果を判断。自己修復できそうなら実行（権限内で）。できなければAgent 4へ
→ kubectl / 再起動 / ロールバック（権限設定次第）
Agent 4
エスカレーション
対処不能と判断したら人間へ。調査レポート付きでSlack通知 / PagerDuty発火
→ Slack API / PagerDuty API

デモ：500エラー発生→エージェントがSlackにレポート

ステージング環境 / 500エラー発生シナリオ
1

怖いのはActだけ。Read OnlyとSuggestは今日から始められる

段階的導入——最初は「調査レポートを作るだけ」から

  • 1
    Read Only フェーズ（リスクゼロ）
    アラートが来たらログを調査してSlackにレポートを送るだけ。人間はそのレポートを見て判断。
  • 2
    Suggest フェーズ（承認フロー付き）
    「再起動を推奨します。実行しますか？ ✅/❌」をSlackで送る。人間がボタンを押すまでは何もしない。
  • 3
    Act フェーズ（ガードレール付き）
    dry-runモードで検証後。blast radius制限あり（1台だけ / タイムアウト5分 / 自動ロールバック付き）。

何から始めるか

STEP 1 — 今週できること

runbookをエージェントが読める形に整理する
手順書が存在するなら、エージェントへの指示書に変換できる。「ログのどこを見るか」「判断基準は何か」を文章で書き出す。

STEP 2 — その次

Read Onlyエージェントを1本作る
アラートWebhookを受けてログ調査→Slackに送るだけ。壊れない。失うものはない。

まとめ

人間が寝てる間に、初動は終わってる

✅ 初動対応の1〜3（調査・確認・報告）はルーティン——エージェントに委任できる

✅ 「全自動」は怖い。Read Only → Suggest → Act の順で導入する

✅ PagerDutyは消えない。人間が呼ばれる回数が減るだけ

✅ 今日から始めるならrunbookをエージェントの指示書に変換するだけでいい
